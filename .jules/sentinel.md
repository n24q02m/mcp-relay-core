## 2024-05-18 - Missing Security Headers in Relay Server API
**Vulnerability:** The Express API (`packages/relay-server`) lacked essential HTTP security headers like `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, and `Strict-Transport-Security`.
**Learning:** Despite the unique zero-knowledge architecture (where sensitive data is encrypted by the client and not readable by the server), defense in depth remains important. Even opaque APIs could be subject to MIME sniffing or clickjacking if served without appropriate headers.
**Prevention:** Always use the `helmet` package to apply a baseline set of security headers on Express servers, even those not directly serving HTML pages.## 2025-02-14 - Type Confusion in Express Request Bodies
**Vulnerability:** Express `.json()` middleware parses request bodies dynamically, meaning fields expected to be strings can actually be arrays or objects. Direct property access like `.length` on these values can lead to unexpected behavior (arrays return their item count, bypassing string length limits), and Type Confusion vulnerabilities.
**Learning:** Checking truthiness (`!field`) and `.length` is insufficient for request body validation. If a client sends `{"sessionId": ["A", "B", "C"]}`, `.length` is 3, bypassing a 256 char limit check, but causing unintended object serialization or logic errors downstream.
**Prevention:** Always explicitly check the type of user input expected to be a string using `typeof field === 'string'` before performing string-specific operations or validations. Additionally, wrap `JSON.stringify` on user-provided objects in `try-catch` blocks to handle potential circular references.
