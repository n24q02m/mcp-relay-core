## 2024-05-18 - Missing Security Headers in Relay Server API
**Vulnerability:** The Express API (`packages/relay-server`) lacked essential HTTP security headers like `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, and `Strict-Transport-Security`.
**Learning:** Despite the unique zero-knowledge architecture (where sensitive data is encrypted by the client and not readable by the server), defense in depth remains important. Even opaque APIs could be subject to MIME sniffing or clickjacking if served without appropriate headers.
**Prevention:** Always use the `helmet` package to apply a baseline set of security headers on Express servers, even those not directly serving HTML pages.
## 2025-02-18 - Express Type Confusion in Length Validation
**Vulnerability:** Input validation based only on existence checks (`if (!sessionId)`) and length checks (`if (sessionId.length > 256)`) in Express `req.body` leaves endpoints vulnerable to type confusion. If an attacker passes an array or a specially crafted object, the `.length` property evaluates differently or gets bypassed entirely (e.g., bypassing max length restrictions and avoiding expected string operations).
**Learning:** In dynamically typed request bodies, especially JSON bodies parsed by Express, checking for truthiness and `.length` is insufficient because arrays also have a `length` and objects without `length` bypass `length > max` checks.
**Prevention:** Always strictly validate the type of input properties (`typeof field !== 'string'`) before performing length-based validation or storing them, even when TypeScript interfaces (`as { ... }`) are used, since TypeScript types do not exist at runtime.
