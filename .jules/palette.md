## 2024-05-24 - Accessible Dynamic Forms and Status Alerts
**Learning:** When dynamically generating form fields from a schema, implicit requirements (like `field.required !== false`) are read by screen readers if the `required` attribute is set, but sighted users need a visual cue. Adding an asterisk `*` wrapped in `aria-hidden="true"` satisfies visual users without double-announcing for screen readers. Furthermore, dynamically injected status containers must have `role="alert"` for screen readers to automatically announce the update without the user shifting focus.
**Action:** Always pair HTML5 `required` attributes with an `aria-hidden` visual indicator in dynamic form builders. Always apply `role="alert"` (or `aria-live="assertive/polite"`) to dynamically injected notification containers.
## 2024-05-24 - Mode Selectors and Server-Pushed Events
**Learning:** For custom "mode selector" buttons that are mutually exclusive, using standard `<button>` elements with `aria-pressed="true/false"` is often better than trying to implement `role="radio"` and `role="radiogroup"`. The radio roles require implementing complete custom keyboard navigation (arrow keys for selection), whereas `<button>` + `aria-pressed` works with standard Tab/Enter/Space navigation out of the box, ensuring keyboard users aren't trapped or confused. Additionally, when injecting dynamic server messages (like polling responses), wrapping the container in `aria-live="polite"` ensures screen readers announce the incoming messages without interrupting the user. Providing external links should also have `rel="noopener noreferrer"` for security and an `aria-label` detailing that it opens a new tab.
**Action:** Use `aria-pressed` for custom toggle/selector buttons to preserve native keyboard accessibility. Apply `aria-live="polite"` to containers that receive asynchronous server updates. Include context like `(opens in a new tab)` for target `_blank` links.
